## Capacities v2.0 API contract (all reads VERIFIED live; writes spec-only)
- Base: https://api.capacities.io   Auth: `Authorization: Bearer <CAPACITIES_IO_API_KEY>`
  Content-Type: application/json. Token is SPACE-BOUND (no spaceId in requests).
- `GET /space` ÔåÆ { id, title }. Sanity-check the configured space name.
- `GET /space/structures` ÔåÆ { structures:[{ id, title, pluralName, ... }] }.
  Resolve the "Atomic Note" type by title. (In this space its id is
  `04192a07-263d-4748-9c53-be8c34abd327`, but resolve it dynamically ÔÇö do not hardcode.)
- `POST /object/markdown` (10/60s) body { "structureId": <id>, "markdown": <str> }
  ÔåÆ creates the object; the object title comes from a leading `# Heading` in the
  markdown. Read the created object `id` from the response defensively.
- `POST /objects/search` (30/60s) body { "query": "Inbox", "structureIds":
  ["RootPage"], "limit": 5 } ÔåÆ results[].{id,title,structureId}. Match title
  exactly. (Verified: the "Inbox" page id here is `3001c21c-eea7-4bcb-bac4-a8198dab32cd`.)
- `POST /blocks/append` (rate-limited) body { "id": <inbox page id>, "position":
  "end", "blocks": [ { "type": "entity", "entityId": <new note id> } ] } ÔåÆ appends
  a link/reference block to the bottom of the Inbox page.
- Rate-limit header `RateLimit: limit=ÔÇª, remaining=ÔÇª, reset=ÔÇª`; 429 ÔåÆ wait `reset`s.

<acceptance_criteria>

## AC-1: `--upload` creates an Atomic Note
```gherkin
Given a valid YouTube URL, CAPACITIES_IO_API_KEY set, and `claude` on PATH
When `yt --upload <url>` runs
Then the transcript is summarized and a new Atomic Note object is created via
     POST /object/markdown (structureId resolved from the "Atomic Note" type),
     with the video title as its title and the summary as its markdown body
And the created note id is reported on stderr
```

## AC-2: Link appended to the bottom of the Inbox page
```gherkin
Given the new Atomic Note was created
When the upload finishes
Then a link block referencing the note (entity block, position "end") is appended
     to the Page titled "Inbox" (found via POST /objects/search on RootPage)
```

## AC-3: `--upload` implies `--summarize`
```gherkin
Given `yt -u <url>` is run without `-s`
When it executes
Then the transcript is summarized (same pipeline as `-s`) and the summary is
     written to the local `<title>.md` AND used for the Atomic Note
```

## AC-4: Graceful failure
```gherkin
Given the upload fails (CAPACITIES_IO_API_KEY unset, "Inbox" page not found, or a
     non-2xx/rate-limit from any Capacities call)
When `yt -u <url>` runs
Then the upload step shows Ô£ù with a clear stderr message, the local `<title>.md`
     is still written, and the process does not panic
```

## AC-5: Upload appears as a progress step
```gherkin
Given `yt -u <url>` runs in a terminal
When the pipeline executes
Then an "Uploading to Capacities" step appears in the stepped display after
     "Summarizing with claude", showing spinner ÔåÆ Ô£ô (or Ô£ù on failure)
```

  <name>Task 1: Capacities client (space check, structure + page lookup, create note, append link)</name>
  <files>src/main.rs</files>
  <action>
    Add Capacities client functions + structs in main.rs (reuse the existing
    blocking `reqwest::Client`, `serde`, `serde_json`; NO new crate). All calls send
    `Authorization: Bearer <key>` + `Content-Type: application/json`.
    - Config from the already-loaded `~/.config/yt/.env`: `CAPACITIES_IO_API_KEY`
      (required for upload; clear error if unset), `CAPACITIES_IO_SPACE_ID`
      (optional space *name*, sanity check), `CAPACITIES_IO_INBOX_PAGE` (optional,
      default `"Inbox"`).
    - `capacities_get_space(client, key) -> Result<(id,title)>`: `GET /space`.
      If a configured space name is present and != title, `eprintln!` a non-fatal warning.
    - `capacities_structure_id(client, key, type_title) -> Result<String>`:
      `GET /space/structures`, return the `id` of the structure whose `title` ==
      `type_title` (e.g. "Atomic Note"); error if not found.
    - `capacities_create_note(client, key, structure_id, markdown) -> Result<String>`:
      `POST /object/markdown` with `{ structureId, markdown }`; parse and return the
      created object `id` (read defensively; error if absent).
    - `capacities_find_page(client, key, title) -> Result<String>`:
      `POST /objects/search` with `{ query: title, structureIds: ["RootPage"],
      limit: 5 }`; return the id of the exact-title match; error "page not found" otherwise.
    - `capacities_append_entity(client, key, page_id, entity_id) -> Result<()>`:
      `POST /blocks/append` with `{ id: page_id, position: "end", blocks:
      [ { type: "entity", entityId: entity_id } ] }`; `.error_for_status()`.
    - On any non-2xx, include status + a short body snippet in the error; on 429,
      mention `reset` seconds (no retry loop ÔÇö one clear error is enough for MVP).
    Avoid: passing a spaceId (token is space-bound); hardcoding the Atomic-Note
    structure UUID; new dependencies; touching transcript/summarize/comments logic.
  </action>
  <verify>`cargo build` succeeds; `grep` shows `/object/markdown`, `/objects/search`, `/blocks/append`, `/space/structures`, `/space`; no new dep in Cargo.toml.</verify>
